## 0. 术语定稿 (2026-05-30 设计问答锁定)

| 概念 | 定名 | 说明 |
|------|------|------|
| 产品名 | **科研流水线 (Research Pipeline)** | 开源发布名 |
| 顶层容器 | **控制台** (单一) | 整个 app = 一个控制台, 统揽全局 |
| 分组层 | **类别** | 大论文 / 小论文 / 专利 / 项目报告 |
| 流程单元 | **状态** (4 块) | 选题立项 → 实验执行 → 成文 → 投稿发表 |
| 视图 1 | **状态总览** | 看项目在哪个状态 (主视图) |
| 视图 2 | **文档库** | 看/浏览项目文件 (原文档打开器) |

### 4 个状态 (8 阶段合并)
1. **选题立项** = Idea + 验证创新性 + 设计实验
2. **实验执行** = 跑实验 + 评估结果
3. **成文** = 选期刊 + 写稿 + 审稿 (SciWrite+蒸馏评审)
4. **投稿发表** = 投稿 + 跟进修回

## 1. 产品定位

一个**流程/状态驱动**的本地工作台, 管理科研产出 (论文/专利/项目) 从想法到投稿的全流程。

- **为谁**: 先自用 (吕博士 + 师妹), 后开源发布
- **核心价值**: 一眼看到"每个项目走到流程哪一步", 而不是堆文件
- **形态**: 本地动态网页 (OpenClaw 式控制台), 自动刷新, 独立运行

---

## 2. 核心理念 (最重要, v0 错在这)

❌ v0 错误: 做成了"文件打开器 / 素材库" (项目 → 一堆文件)

✅ 正确: **流程/状态驱动** (项目 → 它在流程的哪个状态 → 该状态的内容)

- 中心是【状态/流程】, 文件是末端
- 进入一个项目 = 进入它当前所处的状态
- 从左到右 = 流程推进方向
- 能"操作" (新建项目/工作台), 不只是只读看

## 3. 信息架构

层级 (待第 8 节 Q1 确认):
```
工作台 (Workspace, 可多个)
└── 栏位 (Category: 大论文 / 小论文 / 专利 / 项目报告)
    └── 项目 (Project: 如 paper_iad_factory)
        └── 状态/流程 (8 个阶段, 见下)
            └── 文件 (该阶段的内容)
```

### 产品线 (栏位)
1. 大论文 (博士论文)
2. 小论文 (期刊论文)
3. 专利
4. 项目报告

### 项目结构 (2026-05-30 二次更新: 总览 + 文档打开器 双视图)

每个项目 = 【总览】+【文档打开器】两个视图:

#### 视图 A · 总览 (主页, 一屏看全流程, 不细拆)
不再每步一页 (太碎)。一屏展示整个流程, 大致 3 块:
- **前期** = Idea + 验证 + 设计 + 评估 + 选刊 (合在一起, 别拆细)
- **实验** = 跑实验 (第三部分, 突出)
- **写稿 / 投稿** = 放最后

#### 视图 B · 文档打开器 (保留 v0 功能)
浏览/查看该项目的所有文件 (md 渲染等)。
—— v0 那个文件浏览不是错, 是当时"只有它、缺了流程总览"。
   现在它退成"文档打开器"组件, 跟总览互补: 总览看你在哪, 打开器看内容。

界面: 选项目 → 默认看【总览】(流程一屏) → 要看文件切【文档打开器】

(注: 流程具体分几块、各块叫啥, 待吕博士最终确认)

- 主区下: 选中状态的"问题清单 + 文件"

## 6. 技术方案

- **语言**: Python 标准库 (零依赖, 开源友好, `python3 app.py` 即跑)
- **数据源**: 本地文件夹 (项目 = 文件夹, 状态 = 子文件夹, 文件 = md/docx/...)
  - 真源在本地, 不依赖 Notion/外部
- **状态怎么存**: 每个项目一个 `project.json` 记录当前状态 + 各阶段进度
  (或沿用 ledger.json 扩展)
- **前端**: HTML + JS, 轮询 /api 每 5 秒刷新
- **独立运行**: 自包含, 后期 PyInstaller 打包成 exe/app 发布
- **数据位置**: paper_workbench/data/ (或指向 text_workshop/projects 复用现有)
