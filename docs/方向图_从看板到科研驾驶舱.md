# 方向图：从“科研看板”到“个人博士科研驾驶舱”

> 这份文档用于校准 `pipeline-board` 的下一步方向。当前面板已经能回答“我在做什么”，但 v3.1 真正要回答的是：“我今天只需要做哪一个最小动作，怎样启动，做到什么算完成，完成后如何给我正反馈”。

版本：v3.1 Daily Research Cockpit 方向图  
日期：2026-05-31  
适用仓库：`pipeline-board`  
真实内容库：仍由外部内容库承载，不写入公开仓库。

---

## 1. 核心判断

当前 `pipeline-board` 的方向是对的，但还主要长在“看板分支”上：

```text
能看见状态
能看见文档
能看见流程
但还不能真正推动行动
```

v3.1 需要长出来的是“行动驱动主干”：

```text
今日任务
任务飞轮
最小启动动作
完成标准
正反馈
卡住恢复
任务生成与确认
```

因此，v3.1 不应优先做“Agent 展示面板”，而应优先做一个**低压力、强行动、任务飞轮驱动的个人博士科研驾驶舱**。

---

## 2. 树形比喻

```text
外部内容库 = 土壤
  ├─ 真实论文
  ├─ 实验结果
  ├─ 审稿意见
  ├─ 博论材料
  └─ 项目状态

Agent / Prompt / 规则 = 根系
  ├─ 生成任务
  ├─ 拆解任务
  ├─ 生成恢复任务
  └─ 生成正反馈文案

Daily Research Cockpit = 树干
  ├─ 今天做什么
  ├─ 怎么开始
  ├─ 做到什么算完成
  └─ 完成后如何继续

四个主要分支
  ├─ 小论文推进：优先投出 / 改稿 / 重投 / 写稿
  ├─ 实验并行：后台跑实验 / 结果回收 / 补实验
  ├─ 博论维护：每周维护主线 / 综述 / 框架
  └─ 当前看板：状态总览 / 文档库 / 阶段进度
```

解释：

- 外部内容库是土壤，保存真实科研内容。
- Agent / Prompt 是根系，负责把材料转成任务。
- Daily Cockpit 是树干，负责把复杂科研压缩成每天能启动的一小步。
- 现有看板不是错的，它是一个分支，但不是主干。
- 真正对抗拖延的是“任务飞轮 + 最小启动动作 + 正反馈”。

## 3. 当前面板与目标系统的差异

| 维度 | 当前 pipeline-board | v3.1 Daily Research Cockpit |
|---|---|---|
| 核心问题 | 我现在在哪个阶段 | 我今天只做哪一小步 |
| 用户感受 | 看得清楚 | 能启动、压力低、有反馈 |
| 数据中心 | state.json | state.json + tasks.json + today.json + done_log.json |
| 页面中心 | 状态流程、文档库 | 今日任务、任务飞轮、卡住恢复 |
| Agent 位置 | 暂无，或容易变成展示卡 | 后台任务生成器，不抢主界面 |
| 抗拖延能力 | 弱 | 强：小任务、启动任务、正反馈、恢复任务 |
| 博论关系 | 目前按项目分类 | 先独立维护，后续映射小论文 case |

## 4. 任务飞轮

任务飞轮的目标：不要让“完成一篇论文”这种大任务直接压到用户面前，而是拆成可以启动的动作。

```text
长期任务池 projects/*/tasks.json
  ↓
候选今日任务：由规则或 Agent 生成
  ↓
用户确认：今天只选 1–3 个
  ↓
今日任务 planner/today.json
  ↓
Starter Task：打开文档 / 写 3 句话 / 列 5 个点
  ↓
Action Task：完成一个小产出
  ↓
完成记录 planner/done_log.json
  ↓
正反馈：你已经推进了
  ↓
下一轮生成更合适的任务
```

任务至少分为四层：

```text
Big Goal：完成一篇论文改稿
Milestone：完成一个审稿人或一个章节
Action Task：完成某条意见回应草稿 / 补一张表 / 写一个小节
Starter Task：打开文档，写 3 句话；或列出 5 个修改点
```

默认显示 `Action Task + Starter Task`，不要直接把 `Big Goal` 放到首页制造压力。

## 5. 首页应该长什么样

v3.1 首页第一屏不应是复杂流程，而应是：

```text
今日科研驾驶舱

今天只推进 1–3 件事。

主任务
- demo_paper：完成一条审稿意见的回应草稿
  为什么今天做：最接近投出，完成后能推动改稿闭环
  最小启动：打开 response.md，写 3 句话
  完成标准：有回应、有修改位置、有下一步判断

保底任务
- demo_thesis：补 100 字框架定义
  为什么今天做：保持大论文主线不断线
  最小启动：打开 outline.md，只写一句定义

最近完成
- 已完成 3 个改稿拆解任务
- 已整理 1 个补实验清单

当前最卡
- 某论文：重投任务未拆解
- 某实验：补实验结果未回收
```

首页原则：

```text
少任务
强启动
有正反馈
不制造羞耻感
不把复杂流程压到第一屏
```

## 6. 小论文与大论文的关系

当前阶段：先分开。

```text
paper profile：选题 → 实验 → 成文 → 投稿
thesis profile：主线 → 综述 → 章节 → 答辩
```

中期再加入：

```text
大论文读取小论文状态摘要
小论文映射到博士论文章节
大论文显示 case 覆盖度
```

长期目标：

```text
博士论文作为总控台，用统一框架统领多个研究 case。
```

现在不强行做复杂联动，先保证小论文能推进、实验能并行、大论文不断线。

---

## 7. Agent 的正确位置

Agent 不应该成为主界面的装饰。Agent 的价值是生成可执行结果。

```text
Agent / Prompt / Claude Code
  ↓
输出结构化文件
  ├─ tasks.json
  ├─ today.json
  ├─ done_log.json
  └─ revision_plan.md / experiment_plan.md
  ↓
pipeline-board 只读展示
```

因此，v3.1 MVP 不必做 Agent 页面。可以先做：

```text
prompts/daily_planner.md
prompts/revision_task_splitter.md
prompts/recovery_task_generator.md
```

面板显示 Agent 生成并经用户确认后的结果。

---

## 8. 一句话目标

`pipeline-board v3.1` 的目标不是“更复杂的看板”，而是：

> 把复杂科研压缩成每天可以启动的一小步，并用正反馈让这个飞轮持续转起来。

最终用户体验应该是：

```text
我打开面板，不是被提醒还有一堆事没做；
而是看到：今天只需要从这里开始，而且我之前已经做得不错。
```
